[#lz-oparl-ingestion]
= OParl- und Drucksachen-Ingestion

Dieses Szenario beschreibt, wie die strukturierten Parlamentsdaten aus dem
amtlichen Ratsinformationssystem in das System gelangen und für Anzeige und Suche
aufbereitet werden. Quelle ist die standardisierte OParl-Schnittstelle der Stadt.

Der Ablauf ist eine Kette über die Datenablage: `scrape-oparl` legt die
Rohdaten ab, `download-paper-files` lädt die verlinkten PDF-Dokumente, und
`generate-paper-assets` überführt die Drucksachen in das ausgelieferte JSON-Format.
Den Abschluss
bildet `index-search` — diese Stufe gehört nicht allein zur OParl-Verarbeitung,
sondern *führt zwei Pipelines zusammen*: Sie indiziert sowohl die hier
aufbereiteten Drucksachen als auch die `session-speeches-*.json` aus der
Video-Pipeline (<<lz-sitzungsverarbeitung,Szenario 6.1>>).

Der Abruf läuft in einem von *zwei Modi*. Beide unterscheiden sich nur darin,
_welche_ OParl-Objekte geholt werden — der weitere Ablauf ist identisch:

* *full* — vollständiger Abruf ab einem Stichdatum (Erstbefüllung / Bootstrap).
* *incremental* — nur neue und geänderte Objekte (regulärer Betrieb).

[plantuml,lz-oparl-ingestion,svg]
....
@startuml
skinparam shadowing false
skinparam activity {
  BackgroundColor #lightyellow
  BorderColor #999999
}

start
:**scrape-oparl** (Deno)\nOParl-API abrufen;
if (Modus?) then (full)
  :alle Objekte ab Stichdatum holen;
else (incremental)
  :nur neue/geänderte Objekte holen\n(Delta);
endif
:OParl-Rohdaten ablegen\n(data/oparl-magdeburg/);

:**download-paper-files** (Deno)\nverlinkte PDF-Dokumente laden;
:**generate-paper-assets** (Deno)\nDrucksachen in JSON-Batches überführen;
note right
  Eingabe: OParl-Rohdaten + PDF-Dateigrößen
  Ausgabe: data/papers/
end note

:**index-search** (Deno)\nDrucksachen + Redebeiträge\nin Typesense importieren;
note right
  Eingaben:
  - **data/papers/** (aus dieser Pipeline)
  - **session-speeches-*.json**
    (aus Video-Pipeline, Szenario 6.1)
  Ziel: Typesense-Collections
end note
:Datenablage und Suchindex aktualisiert;
stop
@enduml
....

== Ablauf im Einzelnen

[cols="1,3,3",options="header"]
|===
| Stufe (Werkzeug) | Eingabe | Ausgabe

| *scrape-oparl* (Deno)
| OParl-API der Stadt; Modus (full/incremental) + Stichdatum
| Rohe OParl-Objekte (Body, Sitzungen, Drucksachen, Dateien,
  Tagesordnungen, Beratungen, Organisationen) in `data/oparl-magdeburg/`

| *download-paper-files* (Deno)
| Datei-Verweise aus den OParl-Daten
| Heruntergeladene PDF-Dokumente

| *generate-paper-assets* (Deno)
| OParl-Rohdaten + heruntergeladene PDFs (für die Dateigrößen)
| Drucksachen-Metadaten als JSON-Batches in `data/papers/`

| *index-search* (Deno)
| `data/papers/` (aus dieser Pipeline) **und** `session-speeches-*.json` (aus der
  Video-Pipeline, <<lz-sitzungsverarbeitung,Szenario 6.1>>)
| Befüllte Typesense-Collections (`papers`, `speeches`)
|===

Die Volltextsuche selbst läuft anschließend clientseitig im Browser direkt gegen
Typesense; sie ist kein eigener Verarbeitungsschritt und daher hier nicht weiter
ausgeführt.

== Fehler- und Sonderfälle

* *OParl-API nicht erreichbar oder inkonsistent:* Der Abruf ist wiederholbar; ein
  erneuter — typischerweise inkrementeller — Lauf holt fehlende oder geänderte
  Objekte nach, ohne den bestehenden Bestand zu verwerfen.
* *Entkopplung über die Datenablage:* Die nachgelagerten Stufen (Aufbereitung,
  Download, Indizierung) arbeiten auf dem abgelegten Bestand und können unabhängig
  vom Abruf erneut ausgeführt werden.
